# 账户系统落地实现方案

> 本方案基于《账户系统设计》（[`账户系统.md`](./账户系统.md)）的设计目标与《落地差距分析》（[`账户系统落地差距分析.md`](./账户系统落地差距分析.md)）的现状盘点，给出分阶段、可落地的工程实现路线。每个阶段尽量自包含：**数据模型 → 接口 → 关键实现 → 存量迁移 → 验收点**。
>
> 阅读建议：先看 §0 总则（尤其 §0.3 语义重定位、§0.4 迁移策略），再按阶段推进。阶段间有依赖，阶段内部子任务可并行。

---

## 0. 总则

### 0.1 设计原则

- **后端是最终安全边界**：格式 / 权限校验后端必做，前端只做体验性校验。
- **渐进式迁移**：保留旧字段（如 `ObjectItem.ownerId`）作为去规范化指针，新表（`project_member`）为真相源，避免大爆炸式重写。新旧鉴权并行期要短、优先级要明确。
- **软删除优先**：用户资源（评论 / 想法 / 申请 / 动态）删除一律状态置 `DELETED`，保留审计链路；账号注销走 `DELETED` + 匿名化，不立即物理删除。
- **敏感操作必留痕**：封禁 / 解封 / 角色变更 / 所有权转移 / 审核等动作，调用方负责埋审计点（`OperationLogService.record`）。
- **对外不泄露存在性**：登录 / 注册 / 找回统一口径，避免账号枚举。

### 0.2 技术约束（现状，需遵循）

- Spring Boot 4.1 / Security 7 / Jackson 3（`tools.jackson.*`）；Kotlin；JPA。
- 业务层为纯 `@Service` 类（**无 interface/impl 拆分**），DTO 与 Service 同文件共存；响应 DTO 在控制器内 `data class` 组织。
- 权限不在 Spring Security 角色层做（`LoginUser.authorities` 为空），统一由 `AccessService` + JWT `role` claim 软判定，控制器入口调用。
- JWT 无状态 + Redis 白名单（`TokenStore`）：access 走 `auth:token:{jti}`，refresh 走 `auth:refresh:{jti}` 一次性消费，`auth:user:{userId}:sessions|refreshes` 作踢人索引。
- 邮箱验证码按 `scene+identity+UA` 绑定，`verifyAndConsume` 原子消费；限流**仅覆盖发送侧**（间隔锁 + 日上限），校验侧无限流。
- 审计日志当前为异步 JSONL 文件（`OperationLogService`），**不可查询**。
- 密码 BCrypt 默认强度（`SecurityEncoderConfig`，无强度参数）。

### 0.3 语义重定位（最关键的认知转变）

设计文档把"项目管理"从**系统角色**移到了**项目内角色**。这是一次大语义切换，所有后续工作的前提：

| 维度 | 现状 | 目标 |
| --- | --- | --- |
| 系统角色 | `SUPER_ADMIN` / `PROJECT_MANAGER` | `SUPER_ADMIN` / `USER` |
| 谁能创建项目 | 系统角色为 `PROJECT_MANAGER` | `USER` 且 `canCreateProject=true` |
| 项目归属 | `ObjectItem.ownerId` 单字段 | `project_member` 表（`OWNER/MANAGER/MEMBER`） |
| 邀请码 | 注册即 `PROJECT_MANAGER` | 授予 `canCreateProject=true`，注册仍为 `USER` |

⚠ **不要再为 `PROJECT_MANAGER` 系统角色写新逻辑**：所有"项目管理"能力都改由 `project_member` 判定。迁移期 `AccessService` 同时识别旧 `ownerId` 与新成员关系。

### 0.4 数据库迁移策略

- **开发**：`spring.jpa.hibernate.ddl-auto=create`，Hibernate 每次启动重建表，新增字段 / 表自动生效（代价：`./gradlew test` 会清空共享开发库，已知，需注意）。
- **生产**：yaml 为 `validate`，Hibernate 不改表。新增列 / 表需手动执行 DDL（见附录 A），上线前一次性应用。
- 已删迁移 SQL（`src/main/resources/db/migration/V*.sql` 已 staged 删除）——本项目不再走 Flyway，附录 A 的 DDL 仅作参考。
- **存量回填写成一次性幂等脚本**（`ApplicationRunner` 或 `@PostConstruct`），不进 Hibernate 自动建表流程。开发库因 `create` 每次清空，回填脚本必须幂等可重跑。

### 0.5 阶段总览

| 阶段 | 主题 | 对应设计 | 阻塞后续 |
| --- | --- | --- | --- |
| 一 | `User` 实体扩展 + 枚举 | §2.1 / §3 / §7 / §10 | ✅ 是 |

**目标**：把 `User` 补齐为设计要求的账号模型，扩 `Role` / `Status` 枚举，为后续所有功能提供字段基础。

`@PrePersist` / `@PreUpdate` 维护 `usernameLower = username.trim().lowercase()` 与 `updatedAt`。

### 1.2 关键实现

- `UserService.createUser`：
  - 邮箱 `trim().lowercase()`（修复现状 `UserService.kt:60-66` 只 trim 不小写）。
  - 用户名走 `UsernamePolicy`（见下）。
  - 密码走 `PasswordPolicy`（阶段三落地，此处先接入）。
  - 写 `usernameLower`，唯一性改为查 `usernameLower`（大小写不敏感）。
- 新建 `service/UsernamePolicy.kt`：
  - 长度 3–32；字符集 `[A-Za-z0-9_-]`；禁止邮箱格式；保留名黑名单（`admin` / `administrator` / `system` / `official` / `root` / `support`）。
  - 大小写不敏感查重（按 `usernameLower`）。

### 1.3 存量迁移（幂等 `ApplicationRunner`，开发库可重跑）

1. 所有现存用户：`role = PROJECT_MANAGER` 保留（过渡），同时 `canCreateProject = true`（保留其创建能力，阶段四再统一切到 `USER`）。
2. 回填 `username_lower`、`created_at`、`updated_at`。
3. 默认 `email_verified_at = null`（现有账号未走验证码确认邮箱的，视为未验证；若产品认定历史账号邮箱已可信，可回填为 `created_at`）。

### 1.4 大小写不敏感唯一的落地

- 方案：归一化列 `username_lower` + `unique=true`（跨 H2 / MySQL 可移植，Hibernate 自动建索引）。
- 邮箱同理：查询统一用 `lowercase()`，`email` 列保持 `unique`（MySQL 默认大小写不敏感，H2 区分——归一化写入可消除差异）。

### 1.5 验收点

- [ ] `User` 含全部新字段；`Role` / `Status` 枚举值齐全。
- [ ] 注册 / 建号时邮箱小写、`usernameLower` 写入、保留名被拒。
- [ ] 存量账号迁移脚本幂等，重跑不报错。
- [ ] `./gradlew test` 通过（注意会清空开发库）。

## 阶段二：项目成员关系与资源归属

**对应设计**：§2.3、§8、§9、§14.4/5/6
**目标**：建立 `project_member` 表，给四类内容资源补用户归属字段；权限判定从单字段 `ownerId` 升级为成员关系。

### 2.1 数据模型

新建实体 `datasource/jdbc/ProjectMember.kt`：

### 2.2 权限判定改造（`AccessService`）

扩展 `ensureCanManage`，在 `ownerId` 之外识别成员关系：

```kotlin
fun ensureCanManage(user: LoginUser, objectItemId: Int) {
    if (user.role == Role.SUPER_ADMIN) return
    val member = projectMemberRepository
        .findByProjectIdAndUserIdAndStatus(objectItemId, user.id, MemberStatus.ACTIVE)
    if (member != null && (member.role == ProjectRole.OWNER || member.role == ProjectRole.MANAGER)) return
    // 过渡兜底：旧 ownerId 仍生效
    val ownerId = objectItemRepository.findById(objectItemId)
        .orElseThrow { ResourceNotFoundException("项目条目不存在") }.ownerId
    if (ownerId == user.id) return
    throw ForbiddenException("无权管理该项目")
}
```

新增：
- `ensureIsOwner(user, projectId)`：转交 / 删除 / 授权管理员专用。
- `ensureCanManageResource(user, authorUserId)`：作者本人 或 项目管理员 或 超管。

> 迁移期 `ownerId` 与 `project_member` 双轨，阶段八下线 `ownerId`。

### 2.3 存量迁移（幂等）

1. 遍历 `ObjectItem`：对每个 `ownerId != null` 的项目，插 `ProjectMember(projectId, userId=ownerId, OWNER, ACTIVE, joinedAt=now)`（先查后插保证幂等）。
2. `ObjectItem.created_by` 回填为 `ownerId`（过渡假设：旧项目的归属管理者即创建者）。
3. 历史匿名评论 / 想法 / 申请：`author_user_id` / `applicant_user_id` 留 `NULL`，标记为"历史匿名"（§14.12），**不自动绑定**到任何账号。

### 2.4 验收点

- [ ] `project_member` 表存在，唯一约束 `(project_id, user_id)` 生效。
- [ ] `ensureCanManage` 对 OWNER / MANAGER 放行；MEMBER 与非成员 403。
- [ ] 四类资源表含归属字段；历史数据 `NULL` 不报错。
- [ ] `JoinApplicationStatus.WITHDRAWN` 可用。

## 阶段三：鉴权流程加固

**对应设计**：§5.1、§5.2、§6.1、§6.2、§6.3、§14.7/8/9/10
**目标**：统一 `account` 登录、落地密码强度、修复改密邮箱绑定、补找回用户名、验证码错误次数限制。纯鉴权层，改动集中、收益最高。

### 3.1 统一 `account` 登录

`AuthService.login` 改造（`AuthService.kt:83`）：

```kotlin
data class LoginRequest(val account: String, val password: String)

fun login(req: LoginRequest): LoginResponse {
    val key = req.account.trim()
    val user = if (key.contains('@')) userService.findByEmail(key.lowercase())
               else userService.findByUsernameLower(key.lowercase())
        ?: throw UsernameOrPasswordErrorException()
    ensureLoginable(user)            // 见 §6.1 状态校验
    if (!passwordEncoder.matches(req.password, user.password))
        throw UsernameOrPasswordErrorException()
    user.lastLoginAt = LocalDateTime.now()
    userService.save(user)
    return issueTokens(user)
}
```

`UserRepository` 加 `findByUsernameLower(String)`。前端字段从 `username` 改 `account`；`/login/email` 暂保留并标注为"邮箱+密码+验证码二次验证"模式（§5.1 二选一，文档化即可，不与统一登录混用）。

`ensureLoginable(user)`：

- `BANNED` → `UserDisabledException("账号已被封禁")`
- `DEACTIVATED` → 提示联系恢复
- `DELETED` → 统一按"账号或密码错误"（不暴露存在性）
- `emailVerifiedAt == null` → 按产品策略：要求先验证邮箱 / 或允许登录但限制写操作（建议前者）

### 3.2 密码强度校验（`PasswordPolicy`）

新建 `service/PasswordPolicy.kt`，集中校验，供注册 / 改密 / 重置复用：

```kotlin
object PasswordPolicy {
    private val WEAK = setOf("12345678", "password", "qwerty12" /* ... */)  // 常见弱密码表
    private val CLASSES = listOf(Regex("[a-z]"), Regex("[A-Z]"), Regex("\\d"), Regex("[^A-Za-z0-9]"))

    fun validate(password: String, username: String? = null, email: String? = null) {
        require(password.length in 8..64) { "密码长度需为 8-64 位" }
        val hit = CLASSES.count { it.containsMatchIn(password) }
        require(hit >= 3) { "密码需包含大写/小写/数字/特殊字符中的三类" }
        require(password !in WEAK) { "密码过于常见" }
        if (username != null) require(!password.contains(username, ignoreCase = true)) { "密码不能包含用户名" }
        if (email != null) require(!password.contains(email.substringBefore('@'), ignoreCase = true)) { "密码不能包含邮箱前缀" }
    }
}
```

接入点：`UserService.createUser`、`AuthService.changePassword`（新密码）、`AuthService.resetPassword`。
`ChangePasswordRequest` 补 `confirmPassword`；`ResetPasswordRequest` 补 `confirmPassword`。校验两密码一致、新旧不同。

### 3.3 修复改密邮箱绑定（安全偏差，§14.9）

`AuthService.changePassword`（`AuthService.kt:214`）当前用客户端 `req.email` 构造验证码 context，未校验归属。修复：

- **忽略 `req.email`**，改用已认证用户的 `user.email` 构造 `CodeContext`。
- `sendVerificationCode` 的 `CHANGE_PASSWORD` 分支改为**登录态下发码**：
  - 接口加 `@AuthenticationPrincipal`，`req.email` 仅作显示确认，发码以 `user.email` 为准。
  - `CodeContext.userId = user.id`，key 绑 `userId`（发码 / 校验两侧一致）。

```kotlin
@Transactional
fun changePassword(userId: Long, req: ChangePasswordRequest, userAgent: String) {
    val user = userService.findById(userId) ?: throw UserNotFoundException()
    if (!passwordEncoder.matches(req.oldPassword, user.password)) throw UsernameOrPasswordErrorException()
    if (req.newPassword != req.confirmPassword) throw ParamErrorException("两次密码不一致")
    if (req.newPassword == req.oldPassword) throw ParamErrorException("新密码不能与原密码相同")
    PasswordPolicy.validate(req.newPassword, user.username, user.email)
    verificationCodeService.verifyAndConsume(
        CodeContext(scene = Scene.CHANGE_PASSWORD, email = user.email, userId = userId, userAgent = userAgent),
        req.emailCode,
    )
    user.password = passwordEncoder.encode(req.newPassword)
    userService.save(user)
    tokenStore.invalidateAllSessions(userId)
    mailService.sendSecurityNotice(user.email, "PASSWORD_CHANGED")
}
```

### 3.4 找回用户名（§6.3）

新增 `Scene.RECOVER_USERNAME`。接口：

- `POST /api/auth/verification-code`（scene=`RECOVER_USERNAME`）：校验邮箱已注册后发码。
- `POST /api/auth/username/recover` `{email, emailCode}`：校验码 → `mailService.sendUsernameReminder(email, user.username)`；响应统一"若该邮箱已注册，已发送邮件"，不暴露存在性。

### 3.5 验证码错误次数限制（§14.10）

`VerificationCodeService`：key 旁加 `verify:{scene}:...:errcount` 计数，`verifyAndConsume` 失败 `increment`，超阈值（如 5）删除验证码 key 并抛错。匹配成功仍原子消费原文 key。

### 3.6 验收点

- [ ] 用户名或邮箱都能登录；大小写不敏感。
- [ ] 弱密码、含用户名密码、<8 位密码在注册 / 改密 / 重置均被拒。
- [ ] **改密时用他人邮箱验证码无法通过**（关键安全回归）。
- [ ] 找回用户名不泄露账号存在性；邮件含用户名。
- [ ] 验证码连续输错超限后作废。

## 阶段四：公开注册与项目创建资格

**对应设计**：§2.2、§4.1/2/3、§14.1/2
**目标**：开放普通用户公开注册（`USER` 角色），重定义邀请码语义为"授予 `canCreateProject`"。

### 4.1 公开注册接口

`AuthService.registerUser`（新增，与 `registerManager` 并存至阶段八）：

```kotlin
@Transactional
fun registerUser(req: RegisterUserRequest, userAgent: String): RegisterUserResponse {
    verificationCodeService.verifyAndConsume(
        CodeContext(Scene.REGISTER, req.email.trim().lowercase(), null, userAgent), req.emailCode)
    UsernamePolicy.validate(req.username)
    if (req.password != req.confirmPassword) throw ParamErrorException("两次密码不一致")
    PasswordPolicy.validate(req.password, req.username, req.email)
    val user = userService.createUser(req.username, req.password, req.email, Role.USER)
        .also {
            it.canCreateProject = false
            it.emailVerifiedAt = LocalDateTime.now()
        }
    userService.save(user)
    return RegisterUserResponse(user.id!!, user.username, Role.USER)
}
```

路由：`POST /api/auth/register`（`SecurityConfig` 加 `permitAll`）。`RegisterManagerRequest` 同步加 `confirmPassword` + 接 `PasswordPolicy`。

### 4.2 邀请码语义切换

- `registerManager` 改为：建 `USER` 账号 → 消费邀请码成功后置 `canCreateProject = true`（而非赋 `PROJECT_MANAGER` 角色）。
- 新增"凭邀请码补授资格"路径：已注册 `USER` 提交邀请码 → 消费成功置 `canCreateProject = true`，`POST /api/auth/create-project-grant`。
- `InviteCode` 可选扩 `granted_create_project`（默认 true），为将来"邀请码仅注册不授权"留口子。

### 4.3 项目创建资格管理（超管侧）

- `POST /api/admin/users/{id}/grant-create-project`、`/revoke-create-project`：置 `canCreateProject`，埋审计（操作者 / 目标 / 原因 / 时间）。
- 创建项目时校验：`SUPER_ADMIN` 或 `canCreateProject == true`，否则 403。

### 4.4 存量迁移（接阶段一）

- 原 `PROJECT_MANAGER` 用户全部转 `USER`，`canCreateProject = true`。
- 已有邀请码 `UNUSED` 不变（消费时按新语义授权）。

### 4.5 验收点

- [ ] 公开注册产出 `USER`，`canCreateProject=false`，`emailVerifiedAt` 写入。
- [ ] 邀请码注册产出 `USER` + `canCreateProject=true`。
- [ ] `canCreateProject=false` 的 `USER` 创建项目被 403。
- [ ] 超管授权 / 撤权可审计。

## 阶段五：项目成员管理业务

**对应设计**：§9.1/9.2、§14.6
**目标**：把加入申请接成成员关系闭环，落地 OWNER / MANAGER / MEMBER 的加入 / 退出 / 移除 / 转交。

### 5.1 接口（新 `ProjectMemberController`，挂 `/api/project/object-items/{id}/members`）

- `GET /api/project/object-items/{id}/members`：列成员（按角色排序）。
- `POST /api/project/object-items/{id}/members`：OWNER 添加 MANAGER / MEMBER（`{userId, role}`），`ensureIsOwner`。
- `DELETE /api/project/object-items/{id}/members/{userId}`：OWNER 移除成员 / 撤 MANAGER。
- `POST /api/project/object-items/{id}/transfer`：OWNER 转交（`{newOwnerId}`，事务内：旧 OWNER→MEMBER 或 LEFT、新成员→OWNER、同步 `ObjectItem.ownerId`）。
- `POST /api/project/object-items/{id}/leave`：MEMBER / MANAGER 主动退出（OWNER 禁止，提示先转交）。

### 5.2 加入申请闭环

- 创建申请：`JoinApplication.applicantUserId = user.id`（登录后才能申请）；同一 `(applicantUserId, projectId)` 已有 `PENDING` 则拒绝重复。
- 申请状态补 `WITHDRAWN`（用户撤回）。
- `JoinApplicationManagementService.accept`（含 `acceptByAdmin`）事务内：申请置 `ACCEPTED` + `upsert ProjectMember(projectId, applicantUserId, MEMBER, ACTIVE)`，幂等（已成员则不重复插）。
- 处理权限：`ensureCanManage`（OWNER / MANAGER / 超管）。

### 5.3 退出 / 移除规则（§9.2）

- MEMBER 退出：`status=LEFT, leftAt=now`，立即失权。
- MANAGER 退出：先撤管理权限（降级或 LEFT）。
- OWNER 禁止直接退出 / 被移除；必须先转交所有权、归档或删除项目。
- 所有变更写历史（软状态 `LEFT / REMOVED` 保留），**不物理删 `project_member` 行**。

### 5.4 验收点

- [ ] 接受申请 → 同事务产生 ACTIVE MEMBER；重复接受幂等。
- [ ] OWNER 转交后旧 OWNER 失去管理权、新 OWNER 获得；`ownerId` 同步。
- [ ] MEMBER 退出后 `ensureCanManage` 对其 403。
- [ ] OWNER 无法直接退出（明确报错）。

## 阶段六：账号生命周期管理

**对应设计**：§10、§11、§14.1
**目标**：封禁 / 解封 / 停用 / 注销流程，最后管理员保护，敏感操作二次确认。

### 6.1 封禁 / 解封（`AdminUserController` 扩展）

- `POST /api/admin/users/{id}/ban` `{reason, endTime?}`：置 `BANNED`，`tokenStore.invalidateAllSessions(id)` 立即下线，埋审计（操作者 / 原因 / 起止）。
- `POST /api/admin/users/{id}/unban`：置 `ACTIVE`，审计。
- 封禁 / 解禁记录：并入审计日志按 `target_type=USER, action=BAN/UNBAN` 查（也可新建 `user_ban_log` 表，按检索频率决定）。

> 现状 `JwtAuthenticationFilter` 只校验 token 白名单、不读 `status`；封禁后必须靠 `invalidateAllSessions` 立即下线，否则要等 access TTL 到期。封禁流程**务必**调用它。

### 6.2 停用 / 注销（§10.2）

- `POST /api/user/deactivate`（用户自助）：置 `DEACTIVATED, deactivatedAt=now`，吊销全部会话；冷静期内可 `POST /api/user/reactivate` 恢复。
- `POST /api/user/delete`（注销）：
  - 预检：若用户为任一项目 ACTIVE OWNER，拒绝并提示先转交 / 归档 / 删除。
  - 置 `DELETED, deletedAt=now`，吊销会话；`username` / `nickname` 匿名化（如 `deleted_<id>`），邮箱置空或保留哈希防重注册（按隐私策略）。
  - 历史评论 / 想法 `author_user_id` 保留但展示名匿名化（不物理删内容）。

### 6.3 最后一个超级管理员保护（§11）

- 角色变更 / 封禁 / 删除前查询 `countActiveSuperAdmin()`；若目标为最后一个 `SUPER_ADMIN` 且操作会使其失权，抛 `ForbiddenException`。
- 覆盖：降级、封禁、删除、停用。

### 6.4 高风险操作二次确认（§11）

- 敏感操作（角色变更、所有权转交、封禁、注销）要求重新输密或额外验证码：
  - 请求体增 `confirmPassword` 或 `emailCode`，service 层 `passwordEncoder.matches(...)` 后再执行。

### 6.5 验收点

- [ ] 封禁后用户当前 access token 立即失效（白名单已清）。
- [ ] 注销前预检拦截"有 OWNER 项目"的注销。
- [ ] 降级最后一个超管被拒。
- [ ] 敏感操作无二次确认被拒。

## 阶段七：防滥用与会话管理

**对应设计**：§5.2、§5.3、§12
**目标**：登录 / 注册 / 重置限流，会话设备列表与单设备登出。

### 7.1 限流组件（`RateLimiter`，基于 Redis）

- `RateLimiter.failAndCheck(namespace, identity, maxFail, windowSeconds, lockSeconds)`：失败计数，超阈值写 lock key。
- 接入：
  - 登录失败：`login:user:{usernameLower}`、`login:ip:{ip}`，阈值 5 / 10min，锁定 15min。
  - 注册 / 重置 / 验证码：按 `ip` 维度限频。
- 锁定期间返回 429 + 剩余秒，**不暴露具体账号**。

### 7.2 会话管理入口

`TokenStore` 扩展：access value 由 `userId` 改存 JSON `{uid, ua, ip, issuedAt}`（或新增 `auth:sess:{jti}` 存元数据）。

- `GET /api/auth/sessions`：返回当前用户全部有效会话（jti、UA 摘要、ip、签发时间），标记当前。
- `DELETE /api/auth/sessions/{jti}`：`invalidateAccess(jti, userId)`，单设备登出。
- `DELETE /api/auth/sessions`：`invalidateAllSessions(userId)`（"退出全部设备"，已有底层能力，补接口）。

### 7.3 验收点

- [ ] 连续登录失败 5 次后 15min 内拒绝，即使密码正确。
- [ ] 会话列表含设备 / UA / IP；删指定 jti 后该 token 失效。
- [ ] 限流命中不泄露账号是否存在。

## 阶段八：审计落库与 controlPassword 下线

**对应设计**：§11、§13、§14.11/12

### 8.1 审计落库

- 新建 `AuditLog` 实体（字段同 `OperationLogService` 当前写文件字段 + 主键 / 索引：`idx(time)`、`idx(operator_id)`、`idx(action)`）。

### 8.2 controlPassword 下线

   - 每个自服务端点增加 `@AuthenticationPrincipal`，去掉 `controlPassword` 校验。
3. 过渡期双轨（JWT 优先，controlPassword 兼容），明确下线日期；下线后 `ObjectItem.controlPassword` 列保留但不再读，后续可删。
4. `SecurityConfig.kt:74` `/api/admin/project/**` 由 `permitAll` 改 `authenticated`。

### 8.3 匿名投稿收紧（§8.3 / §14.12）

- 评论 / 想法 / 加入申请 POST 由 `permitAll` 改 `authenticated`（`SecurityConfig.kt:66-72`）。
- 存量匿名行 `author_user_id / applicant_user_id = NULL`，标记"历史匿名"，公开展示带"匿名"标识，**不可绑定**到新账号。

### 8.4 收尾

- 删除 `Role.PROJECT_MANAGER` 枚举（确认无引用）；`ownerId` 视使用情况决定保留为去规范化字段或删除。

### 8.5 验收点

- [ ] 审计日志可在管理端按条件分页查询。
- [ ] 项目创建 / 评论 / 想法 / 加入申请均需登录。
