# 账户系统设计

## 1. 设计目标

账户系统用于支持项目展示站中的用户注册、身份认证、内容投稿、项目加入及项目管理。系统需要同时满足以下目标：

- 未登录访客可以浏览公开项目、项目动态、评论和想法。
- 注册用户可以管理自己发布的内容和项目加入申请。
- 获得项目创建资格的用户可以创建项目并成为项目拥有者。
- 项目拥有者可以按项目授权其他用户参与管理。
- 超级管理员负责系统维护、账号管理、项目审核和违规内容处理。
- 所有权限判断必须同时考虑系统身份、项目内身份和资源归属，不能只依赖前端隐藏入口。

## 2. 核心概念

### 2.1 系统角色

系统角色用于描述用户在整个平台范围内拥有的权限。

| 系统角色 | 说明 |
| --- | --- |
| `SUPER_ADMIN` | 系统拥有者和维护者，可以管理账号、项目、邀请码、审核及全局配置 |
| `USER` | 完成注册的普通用户，默认只能操作自己的资源和参与的项目 |

超级管理员虽然可以绕过普通项目的归属限制，但仍必须遵守敏感操作确认、审计日志、最后一个管理员保护等安全规则。

### 2.2 项目创建资格

项目创建资格与系统角色分离，可使用用户字段 `canCreateProject` 或独立权限记录表示。

- 普通用户默认没有项目创建资格。
- 超级管理员可以授予或撤销普通用户的项目创建资格。
- 获得资格的用户可以创建项目，创建成功后自动成为该项目的拥有者。
- 撤销创建资格不会自动移除该用户已经拥有的项目。

### 2.3 项目内角色

项目拥有者不是单一的全局角色，而是用户与具体项目之间的关系。同一用户可以是项目 A 的拥有者、项目 B 的管理员和项目 C 的成员。

| 项目角色 | 说明 |
| --- | --- |
| `OWNER` | 项目拥有者，负责项目资料、动态、成员、申请和项目权限管理 |
| `MANAGER` | 项目管理员，由拥有者授权，只能管理被授权的具体项目 |
| `MEMBER` | 项目普通成员，可以查看成员内容并执行普通成员允许的操作 |

建议增加项目成员关系表：

```text
project_members
- id
- project_id
- user_id
- role: OWNER / MANAGER / MEMBER
- status: ACTIVE / LEFT / REMOVED
- invited_by
- joined_at
- left_at

唯一约束：(project_id, user_id)
```

项目权限需要满足以下规则：

- 每个项目只能有一个有效拥有者。
- 项目拥有者只能管理自己拥有的项目，不能操作其他拥有者的项目。
- 项目拥有者可以为自己的一个或多个项目授权项目管理员。
- 项目管理员只能管理已授权项目，不能操作该用户未获授权的项目。
- 项目管理员默认不能转移项目、删除项目、变更拥有者或继续授予管理员，除非后续增加细粒度权限。
- 项目拥有者退出、注销账号或被永久删除前，必须先转移、归档或删除其拥有的项目。
- 超级管理员可以在必要时接管、转移或撤销项目权限，但操作必须写入审计日志。

## 3. 账号数据模型

用户账号至少需要包含以下信息：

```text
users
- id
- username
- email
- email_verified_at
- password_hash
- nickname
- avatar_url
- status
- system_role
- can_create_project
- created_at
- updated_at
- last_login_at
- deactivated_at
- deleted_at
```

### 3.1 用户名规则

- 用户名与邮箱是两个独立字段，不直接把邮箱长期作为公开用户名。
- 用户名不能为空，必须设置最小和最大长度，例如 3～32 个字符。
- 明确允许的字符范围，建议支持字母、数字、下划线和短横线。
- 用户名比较需要明确是否区分大小写，建议唯一性校验不区分大小写。
- 禁止使用系统保留名称，例如 `admin`、`administrator`、`system`、`official`。
- 用户名不能使用邮箱格式，避免登录时与邮箱标识产生歧义。
- 用户名修改需要设置冷却时间或次数限制，并保留必要的历史变更记录。

### 3.2 邮箱规则

- 每个账号必须绑定一个有效且唯一的邮箱。
- 邮箱保存前需要去除首尾空格并转换为小写。
- 邮箱必须通过验证码或验证链接确认，并记录 `emailVerifiedAt`。
- 未验证邮箱的账号不能完成正式注册或登录。
- 邮箱默认不向其他普通用户公开。
- 修改邮箱时必须验证新邮箱，并向旧邮箱发送安全通知。
- 修改邮箱成功后，应根据安全策略吊销其他登录会话并要求重新登录。

## 4. 注册流程

### 4.1 普通用户注册

普通用户注册至少需要提交：

- 用户名。
- 邮箱。
- 邮箱验证码。
- 密码。
- 确认密码。
- 用户协议及隐私政策同意状态。

注册流程：

1. 校验用户名和邮箱格式。
2. 校验用户名、邮箱是否已被占用。
3. 发送注册场景专用的邮箱验证码。
4. 校验验证码、密码和确认密码。
5. 在同一事务中创建账号并记录邮箱验证时间。
6. 注册成功后，账号系统角色默认为 `USER`，项目创建资格默认为关闭。

如果产品确实需要“只填写邮箱注册”，可以先生成不可公开展示的内部用户名，并要求用户在首次使用个人功能前设置正式用户名，不建议直接把邮箱作为长期用户名。

### 4.2 项目创建资格授予

- 超级管理员可以在用户管理界面授予或撤销项目创建资格。
- 授权和撤权都需要记录操作者、目标用户、原因和时间。
- 也可以使用一次性邀请码授予项目创建资格，但邀请码不能直接授予超级管理员权限。
- 邀请码应包含有效期、使用次数、创建者和使用者记录。

### 4.3 注册安全

- 验证码按业务场景隔离，注册验证码不能用于登录、改密码或重置密码。
- 验证码必须设置有效期、发送间隔、每日发送上限和最大错误次数。
- 验证码验证成功后必须原子消费，不能被并发重复使用。
- 注册接口需要进行 IP、邮箱和设备维度的限流。
- 对外响应避免泄露过多账号存在性信息。

## 5. 登录与会话

### 5.1 登录方式

- 登录时使用统一的 `account` 字段，支持输入用户名或邮箱。
- 服务端负责识别账号类型，客户端不需要分别调用用户名登录和邮箱登录。
- 登录前应完成首尾空格清理和邮箱小写规范化。
- 登录失败统一提示“账号或密码错误”，避免暴露用户名或邮箱是否存在。
- 未验证、已封禁、已停用和已删除账号不能正常登录，并返回适合该状态的提示。

邮箱验证码登录如果继续保留，应明确它是以下哪一种方式：

- 邮箱、密码和邮箱验证码组成的二次验证登录；或
- 仅邮箱验证码的免密码登录。

两种模式不能混用，且验证码必须绑定登录场景和目标账号。

### 5.2 防暴力破解

- 对登录接口实施账号和 IP 双维度限流。
- 连续失败达到阈值后临时锁定登录或要求图形验证码。
- 临时登录锁定建议存储在 Redis，不要直接把临时锁定等同于管理员封禁。
- 登录失败日志不得记录用户提交的密码。
- 超级管理员账号建议强制启用二次验证。

### 5.3 会话管理

- 使用短期访问令牌和可轮换的刷新令牌。
- 刷新令牌只能使用一次，刷新后旧令牌立即失效。
- 用户可以查看当前登录设备、退出指定设备和退出全部设备。
- 普通退出登录只吊销当前会话；“退出全部设备”吊销该用户全部访问令牌和刷新令牌。
- 修改或重置密码、封禁账号、停用账号、关键角色变化后必须吊销全部会话。
- 刷新令牌续期时需要重新读取账号状态和系统角色，不能永久信任旧令牌中的权限。

## 6. 密码规则

### 6.1 密码强度

- 密码长度不得低于 8 位，建议引导用户使用 12 位以上密码。
- 密码至少包含大写字母、小写字母、数字和特殊字符中的三类。
- 需要明确最大长度，例如 64 个字符，防止超出密码哈希算法的安全输入范围。
- 禁止使用常见弱密码、用户名、邮箱或其明显变体作为密码。
- 密码只能保存为安全哈希，禁止保存、打印或传输可恢复的明文密码。
- 前后端都可以校验密码格式，但后端校验是最终安全边界。

### 6.2 修改密码

已登录用户修改密码时需要提交：

- 原密码。
- 新密码。
- 确认密码。
- 根据安全策略提供当前账号绑定邮箱的验证码。

修改规则：

- 原密码、新密码、确认密码均不能为空。
- 原密码只校验是否正确，不要求重新满足当前密码强度规则。
- 新密码和确认密码必须一致。
- 新密码不能与原密码相同。
- 新密码必须满足当前密码强度规则。
- 邮箱验证码必须发送到当前账号数据库中已绑定的邮箱，不能信任客户端提交的任意邮箱。
- 修改成功后吊销全部旧会话，并发送密码修改安全通知。

### 6.3 找回用户名

- 用户输入绑定邮箱并完成邮箱验证。
- 系统将对应用户名发送到已绑定邮箱。
- 接口响应尽量统一为“如果该邮箱已注册，我们将发送邮件”，避免被用于枚举账号。
- 找回用户名不会修改用户密码，也不会自动生成临时密码。

### 6.4 重置密码

重置密码需要提交：

- 绑定邮箱。
- 一次性邮箱验证码或重置链接令牌。
- 新密码。
- 确认密码。

重置规则：

- 不需要用户提供原密码。
- 不强制同时提供用户名，邮箱验证已能证明对绑定邮箱的控制权。
- 重置验证码或链接必须有有效期、错误次数上限且只能使用一次。
- 新密码和确认密码必须一致并满足密码强度规则。
- 系统不得通过邮件发送原密码、随机密码或其他明文密码。
- 重置成功后吊销全部会话，并向绑定邮箱发送安全通知。

## 7. 用户资料管理

注册用户可以管理自己的公开资料，包括：

- 用户名。
- 昵称。
- 头像。
- 个人简介。
- 通知偏好。

用户不能自行修改以下安全字段：

- 系统角色。
- 项目创建资格。
- 封禁状态。
- 项目内角色，但可以主动退出允许退出的项目。

用户修改邮箱、密码等敏感信息时需要重新确认身份。所有敏感字段变更应记录时间，必要时记录变更前后的非敏感摘要。

## 8. 评论、想法和其他用户资源

评论、想法、项目投稿、项目动态和加入申请等用户资源必须记录创建者用户 ID，例如 `authorUserId` 或 `applicantUserId`，不能只保存昵称或联系方式。

### 8.1 资源归属

- 普通用户只能编辑或删除自己创建的评论、想法和加入申请。
- 项目拥有者和项目管理员可以在其项目范围内处理评论和申请，但不能冒充作者修改内容。
- 超级管理员可以审核、隐藏或删除违规内容，但必须留下审核记录。
- 删除优先采用软删除，保留必要的审核和审计信息。
- 被封禁用户的历史公开内容默认不自动删除，由管理员按内容违规情况处理。

### 8.2 内容审核

- 新评论、想法和公开项目投稿可以进入 `PENDING` 状态等待审核。
- 内容状态至少包括 `PENDING`、`APPROVED`、`REJECTED` 和 `DELETED`。
- 用户修改已通过审核的公开内容后，应根据审核策略重新进入待审核状态。
- 用户删除自己的内容后，公开接口不得再返回该内容。
- 审核拒绝时可以保存拒绝理由，并决定是否向作者展示。

### 8.3 匿名投稿兼容策略

如果系统继续允许匿名投稿，需要为匿名用户生成一次性的管理凭证，并明确凭证丢失后无法证明资源归属。考虑到账户体系需要支持“管理自己的资源”，推荐以下策略：

- 项目浏览继续允许匿名访问。
- 评论、想法、项目投稿和加入申请调整为登录后提交。
- 如需保留匿名投稿，只作为兼容功能，匿名内容不能自动归属到任意注册账号。

## 9. 项目加入与成员管理

### 9.1 加入申请

- 用户必须登录后才能申请加入项目。
- 加入申请需要记录 `applicantUserId`，联系方式只作为申请补充信息。
- 同一用户对同一项目只能存在一个有效的待处理申请。
- 已经是项目成员的用户不能重复申请。
- 用户可以在申请处理前主动撤回申请。
- 只有项目拥有者、被授权的项目管理员和超级管理员可以处理申请。

申请状态建议包括：

```text
PENDING -> CONTACTED -> ACCEPTED
        -> REJECTED
        -> WITHDRAWN
```

- 接受申请时，应在同一事务中将申请状态更新为 `ACCEPTED`，并创建项目成员关系。
- 重复接受、并发接受和已处理申请必须进行幂等或状态校验。
- 拒绝申请时可以填写拒绝原因。

### 9.2 加入、退出和移除

- 普通成员可以主动退出项目，退出后立即失去项目权限。
- 项目管理员退出前应撤销其管理权限。
- 项目拥有者不能直接退出项目，必须先转移所有权、归档或删除项目。
- 项目拥有者可以移除普通成员或撤销项目管理员权限。
- 移除成员和主动退出都应保留历史记录，避免删除审计链路。
- 用户退出项目不会删除其过去已经公开发布的项目内容，具体展示方式由内容规则决定。

## 10. 账号状态与生命周期

账号状态建议至少包含：

| 状态 | 说明 |
| --- | --- |
| `ACTIVE` | 正常使用 |
| `BANNED` | 被管理员封禁，不能登录或继续操作 |
| `DEACTIVATED` | 用户主动停用，可按产品规则恢复 |
| `DELETED` | 已注销或进入最终删除状态 |

邮箱是否验证建议通过 `emailVerifiedAt` 单独表示，不与账号封禁状态混合。

### 10.1 封禁

- 封禁需要记录原因、操作者、开始时间和可选的结束时间。
- 封禁生效时立即吊销账号全部会话。
- 被封禁用户不能通过刷新令牌继续登录。
- 管理员应可以解除封禁，解除操作同样需要审计。
- 是否提供用户申诉入口由产品需求决定。

### 10.2 停用和注销

- 用户可以主动停用或申请注销账号。
- 注销前必须检查该用户是否拥有项目；存在拥有项目时必须先完成项目转移、归档或删除。
- 可以设置注销冷静期，在冷静期内允许恢复账号。
- 注销完成后，邮箱和用户名是否可以重新注册需要明确。
- 历史评论、想法和审计记录应按隐私政策进行匿名化、保留或删除。
- 法律、安全和审计要求必须保留的数据不能因普通注销流程被立即物理删除。

## 11. 超级管理员规则

超级管理员可以：

- 查看和管理用户账号。
- 授予或撤销项目创建资格。
- 创建和管理项目创建邀请码。
- 封禁、解封账号并吊销用户会话。
- 审核项目、评论和想法。
- 接管、转移或归档项目。
- 查看必要的操作审计记录。

超级管理员仍受到以下限制：

- 不能查看用户密码、验证码或完整认证令牌。
- 不能删除、降级或封禁系统中的最后一个有效超级管理员。
- 创建超级管理员、角色变更、项目转移和账号封禁等敏感操作必须记录审计日志。
- 高风险操作建议要求重新输入密码或进行二次验证。
- 管理员接口也必须经过后端权限校验，不能只依赖前端页面控制。

## 12. 安全与隐私

- 密码使用 BCrypt、Argon2 等成熟算法进行不可逆哈希，并设置合理成本参数。
- JWT 载荷中不存放密码、邮箱验证码、联系方式等敏感信息。
- 邮箱、联系方式等个人信息默认仅向本人和确有业务需要的项目管理者展示。
- 日志不得记录密码、验证码、完整令牌和不必要的个人信息。
- 验证码、登录、注册、重置密码、评论和加入申请接口都需要限流。
- 修改邮箱、密码、角色和项目所有权时发送安全通知。
- 所有涉及用户输入并公开展示的内容需要防止 XSS，并执行长度和内容校验。
- 文件上传需要校验文件类型、大小和访问权限，用户只能删除自己上传或有权管理项目下的文件。
- API 对资源不存在和无权限的返回应避免泄露私有项目或用户信息。

## 13. 审计日志

以下操作必须记录审计日志：

- 登录成功、登录失败、退出登录和全部设备退出。
- 密码修改、密码重置、邮箱修改和用户名修改。
- 项目创建资格授予与撤销。
- 系统角色变更。
- 账号封禁、解封、停用和注销。
- 项目创建、删除、转移和归档。
- 项目管理员授权与撤销。
- 项目加入申请的接受和拒绝。
- 评论、想法和项目的管理员审核操作。

审计日志至少记录：

- 操作者用户 ID。
- 操作类型。
- 目标类型和目标 ID。
- 操作结果。
- 失败原因的非敏感摘要。
- 时间、IP 和必要的设备信息。

审计日志不得记录密码、验证码或完整令牌，并且普通用户不能修改或删除审计日志。

## 14. 与现有系统的迁移要求

当前系统只有 `SUPER_ADMIN` 和 `PROJECT_MANAGER`，并以项目 `ownerId` 和 `controlPassword` 进行部分项目管理。升级账户体系时需要完成以下迁移：

1. 将普通账号的默认系统角色调整为 `USER`。
2. 增加普通用户公开注册接口。
3. 增加 `emailVerifiedAt`、账号生命周期和项目创建资格字段。
4. 建立项目成员关系表，将现有 `ownerId` 迁移为 `OWNER` 关系。
5. 为评论、想法、加入申请、文件等资源增加用户归属字段。
6. 接受加入申请时同步创建项目成员关系。
7. 将用户名登录和邮箱登录统一为一个账号登录入口。
8. 在后端统一实施密码强度和确认密码校验。
9. 修改密码时只允许使用当前用户绑定邮箱进行验证。
10. 为验证码增加错误次数限制，并使用原子校验和消费。
11. 明确 `controlPassword` 的迁移方案；推荐在账号权限体系稳定后废弃，避免绕过个人授权、撤权和审计。
12. 对已有匿名评论、想法和加入申请保留为“历史匿名内容”，不能未经证明自动绑定到新用户。

迁移期间可以暂时兼容旧接口，但新旧鉴权方式必须有明确优先级和下线时间，避免长期存在两套相互绕过的权限体系。
