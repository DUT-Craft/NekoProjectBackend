# 账户系统落地差距分析与补充建议

> 本文档对照 [`账户系统.md`](./账户系统.md) 的设计目标，审阅当前后端实现（`src/main/kotlin/fun/utf8/nekoprojectbackend/`）后给出差距清单与推进建议。
>
> 结论：设计文档本身较完整，但当前实现仍停留在「两角色（`SUPER_ADMIN` / `PROJECT_MANAGER`）+ `controlPassword` 自服务」的旧体系，设计中的大部分内容尚未落地。下面按优先级列出需要补充的工作，并在末尾给出建议的推进顺序。

---

## 2. 需要补充的核心缺口

### 2.1 🔴 P0 — 数据模型层缺失（阻塞后续所有功能）

#### 2.1.1 `User` 实体字段严重不足（对照设计 §3）

当前 `User.kt` 仅有 `id / username / password / email / nickname / status / role`，缺失：

- `emailVerifiedAt`：设计 §3.2 要求记录邮箱验证时间。当前验证码用完即弃，**邮箱是否验证过无从得知**。
- `avatarUrl`、个人简介、通知偏好（设计 §7）。
- `canCreateProject`：设计 §2.2 的「项目创建资格」独立字段。
- `createdAt / updatedAt / lastLoginAt / deactivatedAt / deletedAt`：生命周期时间戳。
- `status` 只有 `ACTIVE / BANNED`，缺 `DEACTIVATED / DELETED`（设计 §10）。
- `role` 只有 `SUPER_ADMIN / PROJECT_MANAGER`，**没有 `USER`**（设计 §2.1、§14.1）。

#### 2.1.2 没有 `project_members` 项目成员关系表（设计 §2.3、§9）

项目归属目前只是 `ObjectItem.ownerId` 一个字段，无法表达 `OWNER / MANAGER / MEMBER` 三级角色，也无法记录 `invited_by / joined_at / left_at / status`。这是 §14.4 的迁移核心。

#### 2.1.3 用户资源全部缺归属字段（设计 §8、§14.5）

| 实体 | 现状 | 缺失字段 |
| --- | --- | --- |
| `ObjectItemComment` | 只有 `nickName` | `authorUserId` |
| `Mind` | 只有 `nickName / mcId` | `authorUserId` |
| `ObjectItemUpdate` | 无作者字段 | `authorUserId` |
| `JoinApplication` | 只有 `nickName / mcId / contact` | `applicantUserId` |
| `FileRecord` | ✅ 已有 `uploaderId` | — |

由于四类核心内容都没有用户归属，设计 §8.1「用户只能编辑/删除自己创建的内容」目前**根本无法实现**。

### 2.2 🔴 P0 — 鉴权流程缺口

#### 2.2.1 没有普通用户公开注册（设计 §4.1）

当前只有 `POST /api/auth/register/manager`（邀请码注册 PM），`AuthService.registerManager` 内硬编码 `Role.PROJECT_MANAGER`。缺一个默认 `USER` 角色、无需邀请码的公开注册入口。

#### 2.2.2 登录不是统一 `account` 字段（设计 §5.1、§14.7）

`AuthService.login` 只按 `username` 查；`loginByEmail` 又要求 `account` 必须等于该邮箱归属的用户名。设计要求的「输入用户名或邮箱都能登录」未实现。

#### 2.2.3 密码强度零校验（设计 §6.1、§14.8）

`UserService.createUser`、`AuthService.changePassword`、`AuthService.resetPassword` 都直接 `encode(req.password)`，**没有任何长度 / 复杂度 / 弱密码 / 确认密码校验**。这是 §6.1 明确要求、§14.8 列出的迁移项。

#### 2.2.4 缺失的安全流程

- 没有「找回用户名」接口（设计 §6.3）。
- 没有修改邮箱流程（设计 §3.2、§7）。

### 2.3 🟠 P1 — 一个实际的安全偏差

#### 2.3.1 改密时的邮箱验证码没有绑定当前账号（违反设计 §6.2、§14.9）

`AuthService.changePassword`（`AuthService.kt:214`）：

```kotlin
verificationCodeService.verifyAndConsume(
    CodeContext(
        scene = Scene.CHANGE_PASSWORD,
        email = req.email,   // ← 客户端传入，未校验归属
        userId = null,
        // ...
    ),
    req.emailCode,
)
```

`req.email` 来自请求体，但全程**没有 `req.email == user.email` 的校验**。设计 §6.2 明确：「邮箱验证码必须发送到当前账号数据库中已绑定的邮箱，不能信任客户端提交的任意邮箱。」

**修复方向**：忽略 `req.email`，直接用已认证用户的 `user.email` 构造 `CodeContext`；`sendVerificationCode` 的 `CHANGE_PASSWORD` 分支改为登录态下发码、key 绑 `userId`（发码接口本身是匿名的，客户端传来的 userId 无法核验归属，强绑会导致发码/校验两处 key 不一致）。

### 2.4 🟠 P1 — 防滥用与生命周期

#### 2.4.1 登录 / 注册 / 重置接口无限流（设计 §5.2、§4.3、§12）

限流目前**只覆盖验证码发送**（`VerificationCodeService.checkAndRecordSend`）。`/api/auth/login`、`/login/email`、`/reset-password`、`/register/manager` 均无 IP / 账号维度的失败计数与锁定。设计 §5.2 要求连续失败达阈值后临时锁定（存 Redis，区别于管理员封禁）。

#### 2.4.2 会话管理缺前端入口（设计 §5.3）

`TokenStore` 已有 `invalidateAllSessions`，但**没有暴露**「查看登录设备 / 退出指定设备 / 退出全部设备」的接口，目前只在改密 / 重置时内部调用。

#### 2.4.3 账号封禁 / 停用 / 注销流程未实现（设计 §10、§11）

- 没有封禁接口（记录原因 / 操作者 / 起止时间）。
- 没有「最后一个超级管理员」保护（设计 §11）。
- 没有停用 / 注销冷静期、没有「注销前检查是否拥有项目」（设计 §10.2）。
- 高风险操作没有重新输密 / 二次确认（设计 §11）。

### 2.5 🟡 P2 — 设计文档本身可补充的点

以下反过来是**设计文档没充分覆盖现有代码现实**的地方，建议在设计文档里补清楚。

#### 2.5.1 `controlPassword` 的迁移路径太笼统（设计 §14.11）

文档只说「推荐废弃」，但当前 `/api/admin/project/**` 整条自服务链路（`SecurityConfig.kt:74` `permitAll` + service 层验 `controlPassword`）都依赖它。建议在设计文档补充：废弃前需要把哪些自服务操作迁移到 JWT + `ensureCanManage`、过渡期是否双轨、下线判据是什么。

#### 2.5.2 邀请码的语义需要重新定义（设计 §4.2 vs 现状）

现在邀请码 =「注册即 `PROJECT_MANAGER`」（`AuthService.kt:200`）。但设计 §2.2 把「项目创建资格」独立成 `canCreateProject`，§4.2 又说「邀请码授予项目创建资格」。三者关系需要文档讲清楚：是 **邀请码 → `canCreateProject=true` → 创建项目时再建 `OWNER` 关系**？还是保留「邀请码 = 注册 PM」？这直接决定迁移步骤 1 / 3 / 4 的先后。

#### 2.5.3 匿名投稿的兼容策略要明确时间线（设计 §8.3）

当前 `SecurityConfig.kt:66-72` 对新项目 / 想法 / 评论 / 加入申请仍是 `permitAll` 匿名提交。设计 §8.3 倾向「调整为登录后提交」。建议设计文档给出：哪些资源先收紧、存量匿名内容如何标记为「历史匿名」（设计 §14.12）。

#### 2.5.4 审计日志缺管理端查询能力（设计 §11）

`OperationLogService` 写文件，但 §11 要求超级管理员「查看必要的操作审计记录」。文件型存储不易检索，需要决定：是否落库（新建 `audit_log` 表）、是否补一个分页查询接口。

#### 2.5.5 用户名 / 邮箱规则未在注册侧落地（设计 §3.1、§3.2）

设计 §3.1 要求保留名、字符集、大小写唯一性、修改冷却。当前 `UserService.createUser` 只 `trim()` + 查重；邮箱也没有 `lowercase()` 归一化（`UserService.kt:60-66` 只 trim 不小写，设计 §3.2 要求小写）。

## 3. 建议的推进顺序

按依赖关系，建议分四步推进：

1. **先扩 `User` 实体**：加 `USER` 角色、`emailVerifiedAt`、`canCreateProject`、生命周期字段、`status` 扩枚举。后续所有功能都依赖它。
2. **建 `project_members` 表 + 给四类资源加 `authorUserId / applicantUserId`**：数据归属是一切权限的前提。
3. **补密码强度校验 + 统一 `account` 登录 + 修复改密邮箱绑定**：纯鉴权层，改动集中、风险收益比最高。
4. **再做业务流程**：公开注册、成员管理（`OWNER/MANAGER/MEMBER`）、封禁 / 注销、审计落库与查询、`controlPassword` 下线。
